# Greenfield Agent Instructions

This repository is Greenfield, an open-source, C++20, SDK-first creative application engine.

The Greenfield SDK is the reusable runtime/library developers build apps with. The monorepo contains SDK/runtime code, future Studio work, examples and sandbox apps, tools, docs, and tests.

When modifying this repository, follow the coding standards in `docs/coding-standards.md`.

Before editing, inspect the actual local checkout state and treat it as the source of truth. Keep milestone slices narrow, and do not use remote GitHub state as a substitute for local files.

Important priorities:
1. Prefer readability over cleverness.
2. Use RAII for resource ownership.
3. Follow SOLID principles where they improve clarity and maintainability.
4. Keep functions small and focused.
5. Keep cyclomatic complexity below 10 whenever practical.
6. Prefer descriptive names over abbreviations.
7. Avoid premature abstractions, but keep boundaries clean.
8. Design SDK-first: reusable runtime and library boundaries come before app-specific tooling.
9. Keep UI code independent from SDL, Dawn, WebGPU, FreeType, and other concrete platform/rendering/font providers.
10. Keep render commands renderer-agnostic.
11. Keep platform implementations behind platform abstractions.
12. Keep WebGPU-specific code isolated under its backend/module.
13. Keep SDL-specific code isolated under its platform/startup presenter area.
14. Future platform targets should add providers/backends rather than coupling SDK or UI code directly to concrete implementations.
15. Do not add large frameworks, scripting, ECS, editor systems, or mobile support until explicitly requested.
16. Do not implement WASM support unless explicitly requested for a later milestone, but preserve browser-hosted WebAssembly as a v0.1 target and architecture consideration in docs and boundaries.
17. Do not let widgets directly call WebGPU APIs.
18. Keep public positioning open-source and product-neutral. Do not add private product, company, network-appliance, internal-deployment, or proprietary-positioning references unless the user explicitly requests that direction.

## M4-M6E Scope Guard

M4 work was limited to narrow renderer backend selection and Fast2D diagnostic/headless guardrail documentation.

The sandbox exposes `--renderer=webgpu` and `--renderer=fast2d` through one sandbox app. Do not regress this one-sandbox renderer-selection model or create a second sandbox app for renderer comparison. WebGPU remains the default interactive path. Fast2D remains opt-in and visibly interactive through SDL CPU raster presentation.

Fast2D may consume renderer-neutral render commands, prepare backend-local fill operations, rasterize deterministic filled rectangles with source-over alpha blending, hard-edged rectangular borders, hard-edged rounded fills, hard-edged rounded borders, and intersected nested clipping, defer/count text, and report diagnostics from one headless Control Room frame through `--renderer=fast2d --headless` or `--renderer=fast2d --diagnostic`. M6D added opt-in visible Fast2D sandbox presentation through an SDL CPU raster presenter. M6E added backend-local Fast2D visual parity foundation work and stable first visible raster presentation by keeping the Fast2D SDL window hidden until the first valid raster presentation. SDL CPU-buffer upload and first-present window visibility must stay in SDL platform/presentation or sandbox wiring, not in reusable UI/runtime/render-neutral code.

Future Fast2D visual parity work should remain backend-local unless a tiny render-command vocabulary change is truly necessary. Do not make UI/runtime/render-neutral code branch on renderer backend. Do not make Fast2D the default renderer, and do not add full renderer composition, mixed-surface composition, antialiasing, vector paths, transforms, gradients, visual regression CI, Studio, CLI beyond minimal sandbox flags, Canvas2D, Scene3D, shader tools, dashboards, node graphs, a compositor, retained-mode UI, hot reload, Python bindings, or Skia as part of M6E guardrail/doc work. Do not implement Fast2D text by sneaking FreeType into Fast2D, creating a shared text/font service, adding a temporary debug bitmap font, or moving WebGPU FreeType code into shared code without an explicit future scope and boundary-policy decision.

Current M5 work is limited to export and target foundation vocabulary plus the narrow `templates/cpp-cmake-app` scaffold and its CTest guardrail unless the user explicitly requests more implementation. M5 docs may define host platform, renderer backend choice, app project, app target, build/export target, and browser-hosted WebAssembly as a future target direction. Do not add generated projects, CLI behavior, install/package/export logic, Windows-specific workflows, WASM support, optional Dawn/WebGPU or FreeType behavior, or sandbox runtime changes as part of M5 vocabulary/template work.

Current M6A work is limited to product-quality UI runtime groundwork unless the user explicitly requests a different milestone or slice. The implemented foundation preserves the immediate UI model while adding renderer-neutral `UiId` identity, clearer `UiContext` persistent/per-frame state boundaries, minimal persistent focus state, generalized active-control capture, and per-frame mouse press/release consumption.

Current M6B work is limited to the first stateful controls unless the user explicitly requests a different milestone or slice. Checkbox already existed and was audited as complete enough for M6B. Toggle/Switch was added as a new immediate-mode control. Both controls use `UiId`-keyed persistent boolean state, return `true` only when the current frame changes that state, emit renderer-neutral render commands, and use active-control capture plus per-frame mouse press/release consumption. Different `UiId`s keep independent state; matching names intentionally share the same `UiId`-keyed boolean state. Do not turn M6B work into sliders, tabs, dropdowns, modals, toasts, tooltips, keyboard navigation, text entry, IME, clipboard, selection, accessibility, modal focus traps, a retained UI tree, full event dispatch system, broad control inventory, compositor, mixed-surface composition, Studio, CLI, project generation/export tooling, visible Fast2D presentation, Fast2D text rasterization, shared FreeType/text service, Skia, Python bindings, hot reload, or sandbox/runtime renderer behavior changes unless explicitly requested.

Current M6C work is limited to the first continuous/numeric stateful control foundation unless the user explicitly requests a different milestone or slice. Slider was added as a horizontal immediate-mode control backed by private `UiId`-keyed numeric state in `UiContext`. Slider returns `true` only when the current frame changes its value, emits renderer-neutral track/fill/thumb/label commands, uses active-control capture plus per-frame mouse press/release consumption, supports click-to-set and drag-while-captured behavior, clamps values, and safely handles reversed or degenerate ranges. The existing Control Room sandbox contains one small Slider example for manual visual verification; local screenshot capture was proven as a development workflow artifact, but screenshots are not committed project artifacts or required automated test outputs. Do not turn M6C work into keyboard navigation, text entry, dropdowns, tabs, modals, toasts, tooltips, accessibility, a retained UI tree, full event dispatch system, broad control inventory, compositor, mixed-surface composition, Studio, CLI, project generation/export tooling, Canvas2D, Scene3D, shader tools, visible Fast2D presentation, Fast2D text rasterization, shared FreeType/text service, Skia, Python bindings, hot reload, or sandbox/runtime renderer behavior changes unless explicitly requested.

Current M6F work is limited to narrow keyboard/focus navigation groundwork unless the user explicitly requests a different milestone or slice. `InputState` has platform-neutral per-frame keyboard edge fields for Tab, Shift+Tab, Enter, and Space, and SDL key-down events populate those fields behind the platform boundary. `UiContext` rebuilds focusable registration from the current immediate-mode control encounter order. Button, Checkbox, Toggle/Switch, and Slider register as focusable. Tab and Shift+Tab traverse the current-frame order, persisted focus is stored as a `UiId`, stale focus traversal restarts from the frame edge, focused Button activates on Enter/Space, and focused Checkbox and Toggle/Switch toggle on Enter/Space. Slider is focusable within that traversal model. Do not turn M6F work into text entry, character input, IME, clipboard, selection, accessibility, screen reader semantics, modal focus traps, retained UI trees, full event dispatch, shortcut/keybinding systems, spatial navigation, gamepad navigation, renderer/platform coupling, or renderer-specific focus behavior without an explicit later milestone.

Current M6G work is limited to narrow visible focus styling for the implemented M6F focus behavior unless the user explicitly requests a different milestone or slice. Keep focus semantics and focused-state ownership in `UiContext` and UI control code, not in renderers or platform code. Preserve shared `FocusStyle` configurability on Button, Checkbox, Toggle/Switch, and Slider, including support for overriding or suppressing focus visuals with `FocusVisualKind::None`. Do not hardcode non-overridable focus visuals, and do not expand M6G into a full theme engine, accessibility work, text input, IME, clipboard, selection, shortcut/keybinding systems, retained UI, modal focus traps, spatial/gamepad navigation, renderer-specific focus behavior, Studio, CLI, Canvas2D, Scene3D, Fast2D text, Skia, Python bindings, or hot reload.

Current M6H work is limited to narrow focused Slider keyboard adjustment unless the user explicitly requests a different milestone or slice. `InputState` has platform-neutral per-frame keyboard edge fields for Left and Right in addition to the existing Tab, Shift+Tab, Enter, and Space fields, and SDL key-down events populate those fields behind the platform boundary only for non-repeat Left/Right presses. Per-frame input reset clears those edge fields. Slider remains immediate-mode, focusable, renderer-neutral, and `UiId`-keyed inside `UiContext`. A focused Slider may respond to Left/Right by applying a narrow internal step based on the effective normalized range, clamping to normalized min/max bounds, remaining safe for reversed or degenerate ranges, and returning `true` only when the current frame actually changes the value. Do not turn M6H into key repeat policy, a full shortcut/keybinding system, an input action system, text input, character input, IME, clipboard, selection, accessibility, screen reader semantics, spatial navigation, gamepad navigation, Slider page/home/end behavior, a retained UI tree, full event dispatch, dropdowns, tabs, modals, toasts, tooltips, renderer/platform coupling, renderer-specific focus or input behavior, Studio, CLI, Canvas2D, Scene3D, Fast2D text, Skia, Python bindings, or hot reload without an explicit later milestone.
