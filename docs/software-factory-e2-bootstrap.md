# Software Factory E2 Bootstrap Candidate

Status: E2 preparation only. This workflow creates a disposable, reviewed
local candidate. It does not contact GitHub, configure a destination remote,
push history, install software, change production execution, or modify the
Greenfield source checkout.

## Local candidate workflow

Run the bootstrap from a disposable parent directory with an explicit source
checkout, repository identifier, and canonical UTC timestamp:

```text
tools/extraction/prepare-software-factory-bootstrap.sh \
  SOURCE_REPOSITORY OUTPUT_DIRECTORY SOURCE_REPOSITORY_ID \
  EXTRACTION_TIMESTAMP_UTC
```

For example, `SOURCE_REPOSITORY_ID` may be
`lgardner-dev/greenfield`, and the timestamp must be exactly
`YYYY-MM-DDTHH:MM:SSZ`. The source checkout must be on its explicitly selected
branch, normally `main`; the E1 extractor clones only that branch. The output
path must not already exist and must not be inside the source checkout.

The bootstrap invokes the approved E1 extractor, audits the filtered history,
copies the filter-repo commit map, creates the fixed provenance files, and
commits them once. It then audits the committed result, runs `git fsck
--full`, verifies that no remote is configured, and compares source status,
refs, and remotes before and after the run.

The stdout records are concise `NAME=value` lines. The `FINAL_BOOTSTRAP_HEAD`
record is the actual final commit ID. The committed manifest records
`final_bootstrap_head` as the symbolic ref `HEAD`: storing the literal ID of a
commit inside that same commit would require an impossible cryptographic
fixed point. This preserves the one-commit policy while the summary and Git
itself provide the actual ID.

## Generated evidence

The exact generated-path authority is
[`tools/extraction/bootstrap-generated-paths.txt`](../tools/extraction/bootstrap-generated-paths.txt):

```text
docs/provenance/README.md
docs/provenance/manifest.json
docs/provenance/original-to-filtered-commit-map.txt
docs/provenance/retained-files.txt
docs/provenance/filtered-history-audit.txt
```

The manifest is deterministic JSON with schema
`software-factory-bootstrap-manifest-v1`. It records the source repository
ID, source commit and branch, explicit extraction timestamp, tool and
allowlist hashes, pre-provenance filtered `HEAD`, filtered count, map,
inventory, audit, refs, authors, dates, roots, tags, destination-remote
statements, and the extraction architecture document's source commit.

The provenance commit has exactly this message:

```text
Record software factory extraction provenance
```

Its author and committer are fixed as `Greenfield Extraction
<greenfield-extraction@greenfield.invalid>`, with both dates derived from the
supplied timestamp. No source commit, branch, or tag is created. Surviving
source tags are reported for review; no new tags are created.

The post-provenance audit accepts only the five exact generated paths in the
optional allowlist argument:

```text
tools/extraction/audit-filtered-history.sh FILTERED_REPOSITORY \
  tools/extraction/bootstrap-generated-paths.txt
```

The base retained allowlist and generated allowlist remain separate. Every
generated path must exist at `HEAD`, the commit map must match the local
filter-repo evidence, and generated paths found in earlier commits fail. An
undeclared file under `docs/provenance/` is a path violation.

## Review checklist

Before any destination exists, review both independent candidates and
confirm:

- pre-provenance filtered heads, final heads, refs, tree inventories, commit
  counts, generated bytes, and provenance commit hashes are identical;
- only the selected source branch survives, and every surviving tag is
  understood and approved;
- the five generated files exactly match the generated-path allowlist;
- the map, retained inventory, and audit hashes match their files;
- the manifest parses as deterministic JSON and contains no local filesystem
  paths, credentials, secret values, environment dumps, or deployment data;
- the audit has no path violations, high-confidence secrets, unsafe remotes,
  or unexplained binary and large-blob findings;
- the candidate has no configured remote, no alternate object store, and
  passes `git fsck --full`; and
- the Greenfield source checkout has the same status, refs, and remotes as
  before preparation.

The future destination must be a private, completely empty GitHub repository
named `software-factory`: no README, license, `.gitignore`, initial commit,
branch, tag, or other generated content. The preparation script never checks
this remotely; the reviewer must confirm it separately.

## First push and rollback

The actual first push is a separate human action after review. The exact
future procedure is:

1. Create or select the private, empty `software-factory` destination and
   verify its emptiness and visibility in the GitHub UI.
2. Re-run preparation if the source commit, allowlists, scripts, repository
   ID, or timestamp changes; do not push an unreviewed candidate.
3. In the reviewed candidate only, configure the destination remote and verify
   `git remote -v` and `git status`.
4. Push the selected branch once with `git push --set-upstream origin main`.
5. Push only the surviving tags that the reviewer explicitly approved, using
   their exact tag names. Do not force-push or rewrite the reviewed history.

Those remote configuration and push commands must not be run by the E2
preparation script. Keeping the first push human-controlled makes the
destination's privacy, emptiness, identity, and final review gate explicit;
it also prevents a local tooling error from publishing unreviewed history.

Rollback before the first push is simply deleting the disposable candidate
directory and creating a new one. The Greenfield checkout remains the source
of truth. If a destination has been created but the push is rejected, leave
the empty private destination untouched or delete it through the approved
human process; do not rewrite the Greenfield source or force-push a candidate.

The architecture boundary and retained-history decisions remain governed by
[`software-factory-extraction.md`](software-factory-extraction.md) and
[`software-factory-provenance-plan.md`](software-factory-provenance-plan.md).
